# Plan 10-01: Fix `batch_score` non-force image selection

## Intent

When `force=False`, `handle_batch_score` must consider the same catalog / Instagram key sets as the `force=True` path (subject to `date_filter` / `min_rating`), then rely on the existing `image_scores` pre-filter (lines ~1367–1395) to drop triplets already scored at the current `prompt_version` per perspective. Today the non-force branch calls `get_undescribed_catalog_images` / `get_undescribed_instagram_images`, which select rows **without descriptions**; after bulk describe that set is empty, so the job queues zero units (D-01, D-02).

## must_haves (goal-backward)

1. Non-force `batch_score` with `image_type` `catalog` or `both` does **not** call `get_undescribed_catalog_images` inside `handle_batch_score`.
2. Non-force `batch_score` with `image_type` `instagram` or `both` does **not** call `get_undescribed_instagram_images` inside `handle_batch_score`.
3. With `force=False`, described-but-unscored catalog images still produce non-zero `work_triples` after the existing DB pre-filter when at least one `(key, slug)` lacks a current score for that slug’s `prompt_version`.
4. `pytest` for `test_handlers_batch_score` exits 0.

## Verification criteria

- `rg "get_undescribed_catalog_images|get_undescribed_instagram_images" apps/visualizer/backend/jobs/handlers.py` shows **no** matches inside `handle_batch_score` (file may still import or use those elsewhere; zero matches in `handle_batch_score` body is the bar).
- `uv run pytest apps/visualizer/backend/tests/test_handlers_batch_score.py -q` exits with code 0.

<threat_model>

**ASVS L1:** These changes only alter which local SQLite rows are candidates for an operator batch job. They do not add new query parameters to SQL beyond the existing `force=True`-equivalent SELECTs. **Risks:** negligible (larger intermediate `work_triples` before the existing `image_scores` pre-filter). **Mitigations:** same parameterized `date('now', ?)` and `rating >= ?` patterns as the `force=True` branches.

</threat_model>

</read_first>

<action>

In `handle_batch_score`, inside `if image_type in ('catalog', 'both'):` when `force` is **False**, **delete** the block that extends `images_for_scores` from `get_undescribed_catalog_images(lib_db, months=months, min_rating=min_rating)` and **replace** it with the **same** `SELECT key FROM images` logic as the `force is True` branch immediately above it in the same function: start with `sql = "SELECT key FROM images"`, build `conditions` / `sql_params` for `months` (`date_taken >= date('now', ?)` with param `f'-{months} months'`) and `min_rating` (`rating >= ?`), append `WHERE …` when non-empty, `rows = lib_db.execute(sql, tuple(sql_params)).fetchall()`, then `images_for_scores += [(r['key'], 'catalog') for r in rows]`.

Inside `if image_type in ('instagram', 'both'):` when `force` is **False**, **delete** the block that extends `images_for_scores` from `get_undescribed_instagram_images(lib_db, months=months)` and **replace** it with the **same** logic as the `force is True` branch in that same `if`: if `months` is falsy, `rows = lib_db.execute("SELECT media_key FROM instagram_dump_media").fetchall()`; if `months` is truthy, `rows = lib_db.execute("SELECT media_key FROM instagram_dump_media WHERE created_at >= date('now', ?)", (f'-{months} months',)).fetchall()`; then `images_for_scores += [(r['media_key'], 'instagram') for r in rows]`.

Remove `get_undescribed_catalog_images` and `get_undescribed_instagram_images` from the `from lightroom_tagger.core.database import (` block inside `handle_batch_score` if they become unused there (keep `list_perspectives` and any other names still needed).

Do **not** remove or change the existing `if not force and work_triples:` block that queries `image_scores` for `perspective_slug`, `prompt_version`, `is_current = 1` and drops triplets from `work_triples`.

</action>
