<objective>
Implement a **pure-library aggregation layer** that (1) ranks catalog images by an **explainable weighted aggregate** of **current** `image_scores` across **active** perspectives with **coverage guards**, (2) builds a **style fingerprint** from **per-perspective aggregates**, **score distribution**, and **simple rationale token frequencies** plus **example image keys**, and (3) returns **“what to post next”** suggestions for **unposted**, **coverage-eligible** images with **structured reasons** (high score, cadence gap context, token “underrepresented” heuristic). Expose everything via a **new Flask blueprint** under **`/api/identity/`** following `with_db`, pagination, and error patterns from `api/analytics.py` and `api/scores.py`. **No embeddings, no NLP dependencies** — only SQL, Python counters, and reuse of Phase 7 posting population semantics where needed.
</objective>

<action>
Create **`lightroom_tagger/core/identity_service.py`** with a module docstring referencing **08-CONTEXT.md** decisions **D-40–D-47**.

1. **`_active_perspective_slugs(conn) -> list[str]`** — `SELECT slug FROM perspectives WHERE active = 1 ORDER BY slug`.

2. **`compute_image_aggregate_scores(conn, *, min_perspectives: int | None = None) -> tuple[list[dict], dict]`** — For each `images.key` that has at least one current catalog score:
   - Gather rows from `image_scores` joined with slug filter on active perspectives (or left join and ignore inactive slugs).
   - **`aggregate_score`**: weighted mean with **equal weights** over available active-perspective scores for that image (only perspectives that have a current row count).
   - **`perspectives_covered`**: count of active perspectives with a current score.
   - **`eligible`**: `perspectives_covered >= min_perspectives` where `min_perspectives` defaults per **D-41** (implementer picks `max(1, min(2, ceil(active_count/2)))` or a fixed `2` when `active_count >= 2` else `1` — **document in returned `meta`**).
   - Each item: `image_key`, `aggregate_score`, `per_perspective` list entries with `perspective_slug`, `display_name` (join `perspectives`), `score`, `prompt_version`, `model_used`, `scored_at`, short `rationale` optional for detail endpoint vs list (list may omit long text or truncate server-side with max chars).
   - Return **`meta`**: `active_perspectives`, `weighting: "equal"`, `min_perspectives_used`, `total_catalog_images`, `eligible_count`, `scored_any_count`.

3. **`rank_best_photos(conn, *, limit: int, offset: int, min_perspectives: int | None = None) -> tuple[list[dict], int, dict]`** — Sort eligible images by `aggregate_score` DESC, tie-break `date_taken` DESC, `key` ASC (join `images` for `date_taken`, `filename`, `rating`, `instagram_posted`). Return paginated rows + **total eligible count** + meta from (2).

4. **`build_style_fingerprint(conn) -> dict`** — Compute catalog-wide:
   - **`per_perspective`**: for each active slug, `mean_score`, `median_score` (median in Python), `count_scores`.
   - **`aggregate_distribution`**: histogram buckets e.g. `{ "1-3": n, "4-6": n, "7-10": n }` over **eligible** images’ `aggregate_score` only (document).
   - **`top_rationale_tokens`**: `Counter` over tokenized **current** rationales (catalog scores, active slugs only) per **D-43**; return top **30** as `[{token, count}]`.
   - **`evidence`**: for each active slug, list **up to 3** `image_key` with highest score in that slug among eligible images (or among all scored if none eligible — document fallback).
   - **`meta`**: `tokenization_note`, `perspectives_included`.

5. **`suggest_what_to_post_next(conn, *, limit: int, lookback_days_recent: int = 30, lookback_days_baseline: int = 90) -> dict`** — 
   - Candidates: **eligible** + `instagram_posted = 0`, sorted like best photos, cap `limit`.
   - For each candidate, build **`reasons: list[str]`** and **`reason_codes: list[str]`**:
     - Always include **`high_score_unposted`** when in top tier (e.g. aggregate ≥ p90 of eligible unposted or rank ≤ limit — document).
     - **`cadence_gap`**: use `posting_analytics.get_posting_frequency` or SQL using `posted_dump_media_cte_sql()` to compare recent vs earlier window; if below threshold, add **global** `suggestions_meta.cadence_note` and per-image reason only when combined with strong score (avoid noisy copy).
     - **`underrepresented_theme`**: token frequency comparison **posted vs unposted** rationales per **D-46** (posted set = catalog keys with `instagram_posted = 1` or with validated match — pick one consistent with Phase 7; document).
   - Return `{ "candidates": [...], "meta": { ... } }` with **`empty_state`** message when no candidates.

Use **only parameterized SQL**; no f-strings with user input in SQL. Prefer one or two well-commented SQL queries plus Python reduction for clarity.
</action>

<threat_model>
**ASVS L1 — read-only insights**

- **Threat:** SQL injection via query params — mitigate with typed parsing, clamped ints, and bound parameters only (mirror `analytics.py` ISO date patterns if dates are added later).
- **Threat:** Leakage of deployment paths or internal errors — use generic server error responses; log exceptions without full rationale text at INFO.
- **Privacy:** Rationales may contain location/subject PII — treat like captions (trusted local deployment); avoid echoing full text in logs; truncate long fields in list endpoints if needed.
- **Integrity:** Rankings are **advisory**; document in `meta` that scores reflect model/rubric versions at time of scoring.
</threat_model>
