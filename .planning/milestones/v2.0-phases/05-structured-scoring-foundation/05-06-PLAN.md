<objective>
Replace fail-only `_recover_orphaned_jobs` with checkpoint-aware recovery (D-16, D-17): orphaned `running` jobs whose `metadata.checkpoint` is a dict with `checkpoint_version == 1` transition to **`pending`** and receive an informational log; jobs **without** valid checkpoint keep today’s **failed** outcome with the **existing** user-facing log text unchanged. Emit a Socket.IO event so connected clients can surface recovery to the operator (D-16). Wire the frontend to show a dismissible banner on the Processing page when recoveries occur.
</objective>

<task>
<type>implementation</type>
<files>apps/visualizer/backend/app.py</files>
<action>
Rewrite `_recover_orphaned_jobs(db)` in `apps/visualizer/backend/app.py` as follows (concrete control flow):

1. Import at top of function or module level: `from database import update_job_status, add_job_log, get_job` (already have some — avoid duplicate imports).

2. Collect list `recovered_ids: list[str] = []`.

3. For each row from `get_active_jobs(db)` where `status == 'running'`:
   - `job_id = job['id']`
   - `meta = job.get('metadata') or {}`
   - `ck = meta.get('checkpoint') if isinstance(meta, dict) else None`
   - **Resume branch:** if `isinstance(ck, dict) and ck.get('checkpoint_version') == 1`:
     - Call `update_job_status(db, job_id, 'pending', progress=job.get('progress') or 0, current_step='Recovered after restart')` — if `update_job_status` signature cannot set pending from running, instead use direct SQL `UPDATE jobs SET status='pending', completed_at=NULL` via existing helpers; **must end** with row status **`pending`** (verify with `get_job`).
     - `add_job_log(db, job_id, 'info', 'Recovered after restart; job re-queued with checkpoint.')` — exact message string required.
     - Append `job_id` to `recovered_ids`.
   - **Fail branch (D-17):** else:
     - `update_job_status(db, job_id, 'failed')`
     - `add_job_log(db, job_id, 'error', 'This job was still running when the server restarted. It was marked failed; use Retry if you want to run it again.')` — exact string match to current implementation.
     - Do **not** append to `recovered_ids`.

4. After loop: if `recovered_ids` non-empty and global `socketio` is truthy, call `socketio.emit('jobs_recovered', {'job_ids': recovered_ids})` **broadcast** (default room). If `socketio` is None, skip emit.

5. Keep `print` logging: for each resumed job, `print(f"Recovered orphaned job {job_id}: re-queued pending with checkpoint")` exact substring `re-queued pending with checkpoint`.

6. For failed branch keep existing print style but distinguish: `print(f"Recovered orphaned job {job_id}: marked as failed")` (same as today for no-checkpoint case).
</action>

<threat_model>
**ASVS L1 — orphan recovery**

- **Threat:** Re-running attacker-controlled jobs — recovery only flips rows already in jobs DB; no new job IDs created.
- **Threat:** Socket.IO broadcast leaks job IDs to any connected origin — CORS already `FRONTEND_URL`; `jobs_recovered` payload is non-secret UUIDs; acceptable for L1 local deployment. For shared hosts, restrict origins (existing config).
- **Availability:** Recovery loop is O(n) active jobs; cap n implicitly by practical use.
</threat_model>
