<objective>
Persist deterministic checkpoint state in `jobs.metadata["checkpoint"]` for **batch_describe**, **vision_match**, **enrich_catalog**, and **prepare_catalog** so that after a backend crash, completed work is visible in metadata and can be resumed (JOB-01, D-14, D-15). Use **coordinator-thread-only** writes for parallel `batch_describe` / `prepare_catalog` to avoid `update_job_field` races. Extend `match_dump_media` with resume skip + completion callback so `vision_match` checkpoints per `media_key`.
</objective>

<action>
1. Add `apps/visualizer/backend/jobs/checkpoint.py` defining:
   - `CHECKPOINT_VERSION: int = 1`
   - `def fingerprint_batch_describe(metadata: dict, ordered_pairs: list[tuple[str, str]]) -> str:` — return `sha256` hex of UTF-8 JSON object with keys exactly `image_type`, `date_filter`, `force`, `min_rating`, `max_workers`, `provider_id`, `provider_model`, `pairs` where `pairs` is list of strings `"key|itype"` in order (use `json.dumps(..., sort_keys=True, separators=(',', ':'))` before hashing).
   - `def merge_checkpoint_into_metadata(existing_metadata: dict, checkpoint_body: dict) -> dict:` — returns shallow copy of metadata with `checkpoint` key set to `{"checkpoint_version": CHECKPOINT_VERSION, **checkpoint_body}` without dropping unrelated metadata keys.

3. Document in `checkpoint.py` module docstring the exact `checkpoint` keys per job type (see next task).
</action>

<task>
<type>implementation</type>
<files>apps/visualizer/backend/jobs/handlers.py</files>
<action>
**A. `handle_batch_describe`**

- After `images_to_describe` list is built and `total = len(...)`, compute `fp = fingerprint_batch_describe(metadata, images_to_describe)`.
- Read existing `metadata.get('checkpoint')` from `get_job(runner.db, job_id)`: if dict with `checkpoint_version == 1` and `job_type == 'batch_describe'` and `fingerprint == fp`, preload `processed_pairs: set[str]` from `checkpoint['processed_pairs']` (list of `"key|itype"`). Else start empty set.
- Filter `images_to_describe` to exclude any pair whose `"key|itype"` is in `processed_pairs` (resume).
- Define helper `def record_done(key: str, itype: str):` that adds to set, calls `runner.persist_checkpoint(job_id, {'job_type': 'batch_describe', 'fingerprint': fp, 'processed_pairs': sorted(processed_pairs), 'total_at_start': total})` — use **sorted list** for stable JSON.
- **Sequential path:** after each `_describe_single_image` returns, call `record_done`.
- **Parallel path:** only in the `as_completed` loop (main thread), after handling each future result, call `record_done` for that key — never from worker thread.

**B. `handle_vision_match`**

- Before `match_dump_media`, load checkpoint if `checkpoint_version==1` and `job_type=='vision_match'` and fingerprint matches hash of JSON with keys `threshold`, `weights`, `month`, `year`, `last_months`, `media_key`, `force_reprocess`, `force_descriptions`, `provider_id`, `provider_model`, `max_workers` (same `json.dumps(..., sort_keys=True)` then sha256 hex).
- Pass `resume_processed_keys=set(checkpoint.get('processed_media_keys', []))` into `match_dump_media` (new parameter).
- Pass `on_media_complete=lambda mk: runner.persist_checkpoint(job_id, {...})` updating `processed_media_keys` sorted list and same fingerprint fields.

**C. `handle_enrich_catalog`**

- Checkpoint body: `job_type: 'enrich_catalog'`, `fingerprint` from `total` count + first/last key optional — use simple `sha256` of `json.dumps({'total': total, 'keys': [k for k,...]})` built once after list materialized.
- `processed_image_keys: list[str]` append after each successful loop iteration; persist every **1** completed image (can throttle every 5 if needed — default every 1 for determinism).

**D. `handle_prepare_catalog`**

- Same pattern as enrich: `processed_image_keys` for catalog keys after each successful cache write in coordinator thread only if parallel; if parallel workers, aggregate in coordinator only.

**E. On successful `complete_job` for each handler**, clear checkpoint: `runner.persist_checkpoint(job_id, {})` is wrong — instead merge `checkpoint` key to `null` or remove: implement `clear_checkpoint` in runner that deletes `checkpoint` key from metadata OR set `checkpoint: {"checkpoint_version": 1, "cleared": true}` — **preferred**: set metadata to `{**meta, 'checkpoint': None}` and `update_job_field` — SQLite JSON `null` acceptable.

Acceptance: after complete, `get_job` shows `metadata.get('checkpoint')` is **None** or key absent (grep tests).
</action>

<threat_model>
**ASVS L1 — job metadata**

- **Threat:** Metadata ballooning SQLite jobs DB — mitigate: cap `processed_pairs` logging in UI only; if list exceeds **100_000** entries, stop appending and `fail_job` with message containing `checkpoint too large` (defensive).
- **Threat:** Race on metadata updates — coordinator-only writes for parallel jobs; single-threaded handlers use direct persist.
- **Integrity:** Fingerprints prevent applying checkpoints from different job parameters.
</threat_model>
