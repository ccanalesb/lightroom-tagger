<objective>
Add a **library-db analytics module** with parameterized SQL for (1) posting frequency over time, (2) day-of-week × hour-of-day counts, (3) caption length and hashtag (and optional word) rollups, and (4) paginated **unposted catalog** images. Expose these via a small **Flask blueprint** under `/api/analytics/` using the same `with_db` / JSON error patterns as `api/images.py`. All semantics follow **07-CONTEXT.md** (validated matches, timestamp precedence, UTC disclosure in response metadata).
</objective>

<tasks>
<task>
<type>implementation</type>
<files>lightroom_tagger/core/posting_analytics.py</files>
<action>
Create **`lightroom_tagger/core/posting_analytics.py`** with documented helpers (names at implementer discretion) that accept **`db: sqlite3.Connection`** and optional filter args:

1. **`posted_dump_media_base_sql` concept** — central CTE or view logic: join `instagram_dump_media m` to `matches` where `matches.insta_key = m.media_key` AND `matches.validated_at IS NOT NULL`. Expose a single internal fragment reused by all queries to avoid drift.

2. **Frequency series** — `get_posting_frequency(db, *, date_from, date_to, granularity: Literal['day','week','month']) -> tuple[list[dict], dict]` returning buckets `{bucket_start, count}` sorted ascending and a **`meta`** dict: `timestamp_source` (`created_at`|`date_folder`), `granularity`, `timezone_assumption` (e.g. `"UTC"`).

3. **Heatmap** — `get_posting_time_heatmap(db, *, date_from, date_to) -> tuple[list[dict], dict]` with rows `{dow, hour, count}` for `dow` 0–6 and `hour` 0–23; meta includes `dow_labels` hint for frontend.

4. **Caption / hashtag** — `get_caption_hashtag_stats(db, *, date_from, date_to) -> dict` with at least: `post_count`, `with_caption_count`, `avg_caption_len`, `median_caption_len` (compute median in Python from sorted lengths if SQL is awkward), `top_hashtags: [{tag, count}]` capped (e.g. 30), `avg_hashtags_per_post` (float).

5. **Gap list** — `query_unposted_catalog(db, *, date_from, date_to, min_rating, month, limit, offset) -> tuple[list[dict], int]` using `images.instagram_posted = 0` and the same filter patterns as `query_catalog_images` where practical; return lightweight rows for grid: `key`, `filename`, `date_taken`, `rating`, `thumbnail` path resolution **not** in this module (API layer can reuse images blueprint helpers if needed).

Use **only** parameterized SQL. Handle NULL `created_at` by falling back to `date_folder` in SQL (`COALESCE`) per D-31, and document behavior in docstrings.
</action>

<threat_model>
**ASVS L1 — read-only analytics**

- **Threat:** SQL injection via date or filter params — mitigate with typed parsing + bound parameters only.
- **Threat:** Path disclosure via error messages — use generic `error_server_error` for unexpected exceptions.
- **Privacy:** Captions may contain PII — same deployment assumptions as rest of visualizer (local/trusted network); do not log full captions at INFO.
</threat_model>
