<research_summary>
## Summary

The existing implementation already has most of the building blocks needed for this remediation: a reusable job enqueue API (`JobsAPI.create`), Processing tab query-param routing (`?tab=jobs`), and shared path normalization helper (`resolve_filepath`). The current failure mode comes from fragmented discoverability and a weak backend filepath contract in `_handle_batch_embed_image_inner`, which currently checks raw `images.filepath` directly via `os.path.isfile`.

The standard remediation pattern in this codebase is "add capability where users already initiate jobs" plus "emit explicit, structured diagnostics." `CatalogCacheTab` and `MatchingTab` already start jobs from Processing surfaces and direct users to Job Queue via explicit confirmation copy. The embed flow should align with those surfaces instead of remaining mostly hidden behind a similarity error path.

On backend reliability, the right contract is resolve -> validate -> encode, mirroring existing path-safe behavior in `description_service`. This keeps raw catalog metadata decoupled from runtime filesystem reality and enables actionable preflight failure when mounts are unavailable (for example network shares under `//tnas/...`).

**Primary recommendation:** Implement a dedicated embed launcher in Processing and harden `batch_embed_image` with resolved-path preflight + grouped skip reason payload before planning any additional embedding-dependent features.
</research_summary>

<architecture_patterns>
## Architecture Patterns

### System Architecture Diagram

```text
User action (Processing or modal CTA)
  -> JobsAPI.create('batch_embed_image')
  -> backend handler loads candidate keys
  -> resolve_filepath + preflight sample checks
      -> if storage broadly inaccessible: fail fast with actionable reason
      -> else continue
  -> batch encode images
      -> grouped skip/failure accounting
  -> complete job payload (embedded/skipped/failed + grouped reasons)
  -> frontend Job Queue + similarity surfaces render explicit status/next step
```

### Recommended Project Structure (for this phase)
```text
apps/visualizer/frontend/src/components/processing/
  -> Processing tab surface for embed launcher CTA
apps/visualizer/frontend/src/components/image-view/
  -> fallback embed CTA + direct queue navigation affordance
apps/visualizer/backend/jobs/
  -> embed preflight + grouped skip accounting
lightroom_tagger/core/
  -> shared filepath resolution contract
```

### Pattern 1: Processing-first job launcher
**What:** Place embed action where other jobs are launched and monitored.  
**When to use:** User must discover and repeatedly run the operation.  
**Existing analog:** `CatalogCacheTab` and `MatchingTab` enqueue jobs and prompt queue monitoring.

### Pattern 2: Resolve-before-IO contract
**What:** Normalize storage paths before existence checks and encode calls.  
**When to use:** Catalog stores logical/UNC paths but runtime reads from mounted local volumes.  
**Existing analog:** `resolve_filepath()` usage pattern in data-access and describe-related flows.

### Pattern 3: Structured diagnosis payloads
**What:** Return grouped reason counts instead of only free-form warning logs.  
**When to use:** Jobs may partially complete with many skips.  
**Existing analog:** Matching-related handlers already expose counters for downstream UI interpretation.
</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: "Job ran but did nothing"
**What goes wrong:** Job shows progress but mostly skips due to unreachable files.  
**Why it happens:** Raw path checks happen per-row without an early health check.  
**How to avoid:** Add startup preflight sample and explicit fail-fast threshold.  
**Warning signs:** High skip counts with repeated "file missing on disk" and zero embeddings.

### Pitfall 2: Hidden remediation path
**What goes wrong:** Users only discover embed action after hitting similarity failure.  
**Why it happens:** Trigger lives in fallback-only modal path.  
**How to avoid:** Add first-class Processing trigger and direct queue action after enqueue.  
**Warning signs:** Repeated support questions about "where to run embed job."

### Pitfall 3: Ambiguous degradation
**What goes wrong:** Similarity/pin features appear broken instead of dependency-limited.  
**Why it happens:** Missing embeddings/path failures are not surfaced as actionable state.  
**How to avoid:** Surface explicit "embedding unavailable due to storage/path issues" messaging with next action.  
**Warning signs:** Generic fetch error text where embed dependency is the actual blocker.
</common_pitfalls>

## Validation Architecture

- Validation must include both behavioral and contract checks: UI action discoverability, route affordance, backend path handling, preflight guard behavior, and grouped-reason payload shape.
- Use targeted tests near touched surfaces:
  - Frontend: `ImageDetailModal` tests + Processing tab tests for embed trigger and queue deep-link action.
  - Backend: `test_handlers_batch_embed_image.py` (or equivalent) for preflight and grouped reason accounting.
- Include one regression test ensuring resolved filepath is used for `os.path.isfile`/encode input, not raw DB filepath.
- Treat "fail-fast preflight on inaccessible storage" as blocking behavior, not optional logging.
