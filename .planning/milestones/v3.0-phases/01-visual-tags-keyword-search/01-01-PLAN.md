## Objective

Add nullable `dominant_colors`, `mood_tags`, `has_repetition`, and `description_search_document` columns to `image_descriptions`, create SQLite FTS5 virtual table `image_descriptions_fts` as external content (`content='image_descriptions'`, `content_rowid='rowid'`) with `tokenize='porter unicode61'` and a single indexed column `description_search_document`, backfill existing rows, rebuild the FTS index once at migration time, and maintain FTS rows from Python inside `store_image_description` (catalog rows only get a non-null search document and FTS row; non-catalog rows delete any FTS entry). This is the foundation for VIS-01 storage and NLS-02 search.

<task id="T02" type="execute">
<title>Implement search document builder and extend store_image_description + FTS maintenance</title>
<read_first>
- `lightroom_tagger/core/database.py` — full `store_image_description` function body, `library_write`, `_serialize_json`
- `.planning/phases/01-visual-tags-keyword-search/01-CONTEXT.md` — D-04, D-06, FTS scope: catalog only
</read_first>
<action>
1. Add a module-level function `build_description_search_document(summary: str, subjects_json_or_obj: object) -> str` that:  
   - Normalizes `summary` to a single-line-ish string: strip, collapse internal whitespace runs to one space.  
   - If `subjects_json_or_obj` is already a `list` of strings, join with `' '`. If it is a JSON string, `json.loads` and join string elements; on decode error or non-list, treat subjects as empty.  
   - Return `summary_part` if no subjects; else `f"{summary_part} {joined_subjects}"` with one space between parts (D-06: summary + flattened subjects).

2. Extend `store_image_description` SQL to include columns `dominant_colors`, `mood_tags`, `has_repetition`, `description_search_document` in both INSERT and `ON CONFLICT ... DO UPDATE` lists. Bind values:  
   - `dominant_colors` / `mood_tags`: `_serialize_json(record.get('dominant_colors'))` and `_serialize_json(record.get('mood_tags'))` when value is list/dict; if key missing use `None` (NULL in SQL).  
   - `has_repetition`: coerce stringly-typed model output as `1 if record.get('has_repetition') in (True, 1, '1', 'true', 'yes') else (0 if record.get('has_repetition') is not None else None)` before binding; only 0, 1, or NULL in SQL.  
   - `description_search_document`: if `record.get('image_type') == 'catalog'`, set to `build_description_search_document(record.get('summary',''), record.get('subjects', []))`; else NULL.

3. Still inside the same `with library_write(db):` block, **after** the upsert completes, maintain FTS with SQLite-supported DML only (do **not** use `INSERT … VALUES('delete', …)` on FTS5 — it is invalid for external-content FTS5 and can corrupt or error):  
   - `row = db.execute("SELECT rowid FROM image_descriptions WHERE image_key = ?", (record['image_key'],)).fetchone()`  
   - If `row` is not None: `rowid = row[0]`; `db.execute("DELETE FROM image_descriptions_fts WHERE rowid = ?", (rowid,))`  
   - Let `doc` be the bound `description_search_document` text for this write (same value as stored on the row for catalog, else empty/NULL). Only if `record.get('image_type') == 'catalog'` and `doc` is a non-empty string: `db.execute("INSERT INTO image_descriptions_fts(rowid, description_search_document) VALUES(?, ?)", (rowid, doc))`  
   - If non-catalog or document NULL/empty: omit the insert so the FTS row stays removed after the DELETE.

4. Do not add SQL triggers; all FTS writes stay in this function (D-04).
</action>
