# Phase 5.2 Plan 02: Capability detection Summary

Added model capability surfacing (`tool_calling`) end-to-end and updated Search chat model selection to only use tool-capable models.

## Tasks Completed
- **Task 1** - Provider capability metadata added in config (`providers.example.json`) and surfaced via `ProviderRegistry.list_providers()`.
- **Task 2** - `GET /api/providers/models/description` now includes `tool_calling` per model entry.
- **Task 3** - Frontend model typing (`DescriptionModel`) extended with `tool_calling`; Search chat selectors now derive from `chatModels = allModels.filter(m => m.tool_calling)`.

## Delivered Behavior
- Chat UI now excludes non-tool-capable models from provider/model selectors.
- If no tool-capable models are available, Search page shows a clear disabled-state message and blocks submit.
- Selected provider/model persistence still works, but now constrained to tool-capable options.

## Deviations from Plan
- Capability flags were applied in `providers.example.json` (tracked config template) rather than only `providers.json`.
- Final provider set includes `openrouter` as tool-capable in this repository state.
