# Phase 5.2 — Tool-calling search

**Status:** Pending  
**Added:** 2026-04-24  
**Goal:** Replace the single-shot JSON filter LLM with a multi-tool architecture where the model reasons about intent and calls typed tools — enabling superlatives ("the best"), quantity control, and smarter query composition.

---

## Context & motivation

The current approach sends a system prompt describing all filter fields and asks the LLM to return one JSON object. This breaks for:
- Superlatives: "THE best" returns 622 results instead of 1 (no `limit` lever)
- Invented field values: LLM makes up `score_perspective: "overall"` when it doesn't exist
- No runtime discovery: LLM can't ask "what perspectives are available?" before deciding

Tool calling lets the model reason end-to-end — it chooses which tools to invoke, with what params, after understanding the user's intent.

---

## Design decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Tool granularity | Many focused tools | Composes naturally; model picks only what it needs |
| Capability detection | Filter model selector to tool-capable models only | Better UX than silent fallback; JSON approach remains for non-chat features |
| Tool result payload | Rich metadata, bounded by model-requested `limit` | Model needs description/score to give a useful response; user controls context via "next 10" etc. |
| Multi-turn history | Store tool_call + tool_result turns in conversation | Model retains search context across turns |

## Tools to implement

### `search_catalog`
Main workhorse. All params optional; model sets only what it needs.
```json
{
  "name": "search_catalog",
  "description": "Search the photo catalog with optional filters. Returns images sorted by relevance or score.",
  "parameters": {
    "description_search": "string — what's visually in the photo (people, scenes, colors, mood)",
    "score_perspective": "string — perspective slug (use get_scoring_perspectives first)",
    "sort_by_score": "'asc' | 'desc'",
    "sort_by_date": "'newest' | 'oldest'",
    "min_score": "integer 1–10",
    "min_rating": "integer 1–5 (Lightroom star rating)",
    "limit": "integer — how many results to return; use 1 for 'the best', 10 for browsing"
  }
}
```

### `get_scoring_perspectives`
Runtime discovery — model calls this when user mentions scoring/quality before calling `search_catalog`.
```json
{
  "name": "get_scoring_perspectives",
  "description": "Returns available scoring perspective slugs and their descriptions.",
  "parameters": {}
}
```

### `filter_by_date`
Dedicated temporal queries.
```json
{
  "name": "filter_by_date",
  "description": "Filter images by date range.",
  "parameters": {
    "date_from": "YYYY-MM-DD",
    "date_to": "YYYY-MM-DD",
    "sort_direction": "'newest' | 'oldest'",
    "limit": "integer"
  }
}
```

## Tool result payload

Each tool returns:
```json
{
  "total_matched": 622,
  "returned": 1,
  "images": [
    {
      "key": "abc123",
      "filename": "DSC001.jpg",
      "date_taken": "2023-05-14",
      "score": 9.1,
      "score_perspective": "street",
      "score_rationale": "Strong diagonal composition with saturated red element...",
      "description": "A red telephone booth on a rainy London street...",
      "mood_tags": ["melancholic", "urban"],
      "dominant_colors": ["#cc2222", "#888888"]
    }
  ]
}
```

`total_matched` lets the model say "I found 622 — here's the top 1. Ask for more?"

## Capability detection

- Backend: `GET /api/providers/models/description` gains a `tool_calling` boolean per model entry
- Detection strategy: check provider config flag first; if missing, probe with a minimal tool call and cache the result
- Frontend: model selector in chat search marks or hides non-capable models with a tooltip

---

## Multi-turn history format

Conversation stored as:
```
user: "gimme the best red street photo"
assistant: [tool_call: search_catalog({description_search:"red", score_perspective:"street", sort_by_score:"desc", limit:1})]
tool: [result: {total_matched:622, returned:1, images:[...]}]
assistant: "Here's your top street photo with red — scored 9.1. Shot in London 2023..."
user: "show me 5 more"
assistant: [tool_call: search_catalog({description_search:"red", score_perspective:"street", sort_by_score:"desc", limit:5, offset:1})]
...
```

## Implementation plan

### Wave 1 — Backend tool infrastructure
1. Define tool schemas (JSON Schema) in a new `lightroom_tagger/core/search_tools.py`
2. Implement tool executor: `execute_tool(name, args, db) → dict`
3. Add `run_tool_calling_search(messages, tools, provider_id, model, db)` in `nl_catalog_search.py` — handles the LLM loop: call → tool_calls response → execute → feed result → final response

### Wave 2 — Capability detection
4. Add `tool_calling: bool` to provider model entries (config + probe logic)
5. Update `GET /api/providers/models/description` to include `tool_calling` flag
6. Frontend: filter/mark models in chat selector based on `tool_calling`

### Wave 3 — API wiring
7. Update `POST /api/images/chat-search` to branch on model capability: tool-calling path vs existing JSON path
8. Tool result + tool call stored in returned `messages` for frontend conversation history

### Wave 4 — Frontend conversation history
9. `SearchPage.tsx`: persist tool_call/tool_result turns in local message history so "next 10" works correctly

---

## UAT criteria

- [ ] "gimme THE best scored red photo in street photography" → exactly 1 result returned
- [ ] "show me 5 more" on the next turn → 5 different results, same filters
- [ ] `get_scoring_perspectives` is called before `search_catalog` when user mentions scoring
- [ ] Non-tool-capable models (e.g. basic Ollama models) are marked/hidden in the chat model selector
- [ ] Multi-turn conversation retains context — follow-up narrows previous results
- [ ] Tool result includes description, score, rationale so assistant response is grounded
