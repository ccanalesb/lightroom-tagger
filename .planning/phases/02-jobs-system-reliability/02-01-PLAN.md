<objective>
Wire cooperative cancellation end-to-end: the job processor’s `JobRunner` instance is reachable from the REST cancel route, each running job registers a per-job `threading.Event`, `DELETE /api/jobs/<id>` sets the event and database status without being overwritten by `complete_job`/`fail_job`, and long-running `vision_match` work checks cancellation between dump-media iterations. This closes the main gap for SYS-02 and stabilizes cancelled/failed terminal states for SYS-01.
</objective>

### Task 2: JobRunner — register Event, terminal-state guards, cancel helpers
**Type:** modify
**Files:** apps/visualizer/backend/jobs/runner.py

<read_first>

- apps/visualizer/backend/database.py
- apps/visualizer/backend/app.py
</read_first>

<action>
1. At top of `runner.py`, add `import threading` and `from database import get_job` (in addition to existing `database` imports).
2. Replace `active_jobs: {}` usage with: on each `start_job(job_id, job_type, metadata)`, after loading the job row via `get_job(self.db, job_id)`, if row is missing or `row['status'] == 'cancelled'`, **return `False`** without calling `update_job_status` to `running`. Otherwise create `cancel_event = threading.Event()`, set `self.active_jobs[job_id] = cancel_event`, then proceed with existing `update_job_status(..., 'running', ...)` and log; **return `True`**.
3. Add `def is_cancelled(self, job_id: str) -> bool:` returning `True` if `job_id in self.active_jobs` and `self.active_jobs[job_id].is_set()`, else `False`.
4. Add `def clear_cancel_registration(self, job_id: str) -> None:` that `self.active_jobs.pop(job_id, None)`.
5. Replace broken `cancel_job` body that calls `.cancel()` on handles: implement `def signal_cancel(self, job_id: str) -> None:` that, if `job_id in self.active_jobs`, calls `self.active_jobs[job_id].set()` (do **not** write DB here — API owns DB transition for predictable ordering).
6. In `complete_job` and `fail_job`, first line body: `row = get_job(self.db, job_id)`; if row and `row.get('status') == 'cancelled'`, call `self.clear_cancel_registration(job_id)` and **return** without changing status/fields.
7. At successful end of `complete_job` and `fail_job` (after existing updates), call `self.clear_cancel_registration(job_id)`.
8. Add `def finalize_cancelled(self, job_id: str) -> None:` that: calls `self.clear_cancel_registration(job_id)`; ensures `update_job_status(self.db, job_id, 'cancelled')` is idempotent; adds log `add_job_log(self.db, job_id, 'info', 'Job stopped after cancel request')` **only if** the latest `get_job` `status` is already `'cancelled'` and logs do not already contain that exact message string (simple check: any log entry with `message == 'Job stopped after cancel request'` skips append); if status is still `running`, set `cancelled` and add the log once.
</action>

<action>
1. Do **not** add a module-level `from app import ...` (risk of import cycle with `create_app`). Inside the `cancel_job` function body, as the first lines, lazy-import: `from app import get_job_runner, socketio`.
2. In `cancel_job` route for `job['status'] == 'running'`:
   - Call `update_job_status(current_app.db, job_id, 'cancelled')` (keep single source of truth order: **DB first**).

4. After cancel success path, emit realtime update: use the lazy-imported `socketio` and `from database import get_job` (top-level import in `jobs.py` is fine), then `if socketio: socketio.emit('job_updated', get_job(current_app.db, job_id))` (mirror `_job_processor` payload shape).
</action>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| TOCTOU race: cancel vs start_job / complete_job | Medium | DB status checks in processor and runner terminal methods; cancel sets DB then event; handlers consult `is_cancelled` before writes. |
| Unauthenticated job cancel (ID guessing) | Medium | No change to auth in this plan — assumes existing session/API exposure matches Phase 1; document follow-up if routes are later made public. |
| Denial of service via cancel spam | Low | Event set is idempotent; DB writes are single-row updates. |
