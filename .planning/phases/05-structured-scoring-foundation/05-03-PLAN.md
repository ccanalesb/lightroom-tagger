<objective>
Add **Pydantic v2** (`model_validate_json` / `model_validate`) and a centralized module `structured_output.py` implementing **D-12 end-to-end for score-shaped payloads**: deterministic JSON repair → validate → **one** optional LLM “JSON-only repair” pass via a `vision_client` helper (strict prompt, no analysis) → raise typed failure if still invalid. Provide golden fixture tests (D-13) proving deterministic repair, **LLM fixer path (mocked API)**, and hard failure. Multi-perspective describe blobs parsed by `parse_description_response` stay on the legacy silent-fallback path until Phase 6+ refactors that pipeline (see 05-RESEARCH.md §3 — explicit non-goal for Phase 5). This gates `image_scores` persistence so malformed score JSON never silently becomes empty rows.
</objective>

<action>
Create `lightroom_tagger/core/structured_output.py` with:

0. **Constants** at module top: `STRUCTURED_OUTPUT_MAX_CHARS = 512_000`; `STRUCTURED_OUTPUT_RAW_PREVIEW_MAX_CHARS = 200` for `StructuredOutputError.raw_preview` truncation (threat model). `repair_json_text` and public parse entrypoints must reject when `len(raw) > STRUCTURED_OUTPUT_MAX_CHARS` by raising `StructuredOutputError` whose message contains exact substring `input too large`.

1. **Models** (Pydantic v2, `model_config` strict optional as needed):
   - `class PerspectiveScorePayload(BaseModel):` fields `analysis: str`, `score: int` with `Field(ge=1, le=10)`.
   - `class ScoreResponse(BaseModel):` fields `perspective_slug: str`, `score: int` Field 1-10, `rationale: str` (this models **one** perspective line item for `image_scores` insert, not the full describe blob).

2. **`def repair_json_text(raw: str) -> str:`** — deterministic only:
   - Strip whitespace; if wrapped in markdown fences ```json ... ``` extract inner (reuse regex idea from `parse_description_response`).
   - Remove trailing commas before `}` or `]` using a single-pass regex or small state machine (document limitation: nested edge cases OK if documented).
   - Return repaired string.

3. **`def parse_score_response(raw: str) -> ScoreResponse:`** — parse only (no LLM): run `repair_json_text`, then `ScoreResponse.model_validate_json(text)`.

4. **`class StructuredOutputError(Exception):`** with optional fields `raw_preview: str`, `validation_errors: list[str]`. Whenever `raw_preview` is set from model output, store **at most 200 characters** (truncate with ellipsis if longer); same limit applies inside `__str__` / message assembly so threat-model previews never exceed 200 chars.

5. **`def parse_score_response_with_retry(raw: str, fixer: Callable[[str, str], str] | None = None, llm_fixer: Callable[[str, str], str] | None = None) -> tuple[ScoreResponse, bool]:`**
   - Returns `(model, repaired_flag)`.
   - Try `parse_score_response(raw)` → return `(model, False)` on success.
   - On `ValidationError`, set `repaired_flag` True if `repair_json_text` changed text and second parse succeeds.
   - If `fixer` is not None, call `fixer(raw, str(exc))` once to get `candidate` str, then parse `candidate`; success → `(model, True)`.
   - If still failing and `llm_fixer` is not None, call `llm_fixer(raw, str(exc))` **once** to get `candidate`, then `parse_score_response(candidate)`; success → `(model, True)` (repaired_flag True).
   - If still failing, raise `StructuredOutputError` with message containing exact substring `Score response validation failed` and include `validation_errors` from Pydantic; set `raw_preview` using the **200-char cap** above.

**Note:** `fixer` is for tests and narrow call-site overrides. **`llm_fixer` is normally `make_score_json_llm_fixer(...)` from `vision_client`** (D-12 second leg). Phase 6 scoring handlers MUST pass a real `llm_fixer` before persisting scores (see must_haves).

6. **Logging hook:** optional module-level or parameter `log_repair: Callable[[str], None] | None` — when repair path succeeds, invoke with exact prefix string `[structured_output] repaired:` followed by one-sentence reason (static prefixes checked in tests).

Export public names via `__all__`.
</action>

<threat_model>
**ASVS L1 — parsing and validation**

- **Threat:** ReDoS or catastrophic backtracking on huge LLM output — mitigate: cap input length (e.g. refuse `repair_json_text` if `len(raw) > 512_000`) and raise `StructuredOutputError` with substring `input too large`; document constant name `STRUCTURED_OUTPUT_MAX_CHARS = 512_000` in module.
- **Threat:** Exception messages leaking secrets — mitigate: `StructuredOutputError.raw_preview` and any logged preview truncate to **200** chars max (enforced in code + tests); never log full API keys.
- **Availability:** Parsing is CPU-only; length cap prevents memory exhaustion from pathological outputs.
</threat_model>
