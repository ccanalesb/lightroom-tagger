# Plan 05: Close UAT gaps — catalog API 500 + hidden filter bar

<objective>
Restore a working Catalog tab experience documented in [01-UAT.md](./01-UAT.md): `GET /api/images/catalog` must return HTTP 200 with image rows for a valid indexed catalog, and the Catalog tab must always show filter controls while distinguishing **empty database**, **filtered-to-zero**, and **failed load** (including prior 500s that left `total === 0`).
</objective>

<tasks>

## Task 1: Investigate and fix the 500 error on `GET /api/images/catalog`

<read_first>
- `.planning/phases/01-catalog-management/01-UAT.md` (Gaps — root causes and artifacts)
- `apps/visualizer/backend/api/images.py` (`list_catalog_images` / catalog route handler)
- `apps/visualizer/backend/utils/db.py` (`with_db` decorator — error handling)
- `lightroom_tagger/core/database.py` (`init_database`, `_migrate_unified_image_keys`, `_migrate_add_column`, `query_catalog_images`)
</read_first>

<action>
Reproduce the failure: run the visualizer backend (or a minimal Flask test client) and call `GET /api/images/catalog` with typical query params; capture stack traces or the JSON `error` body returned by `with_db`.

Trace the exception to its source:

- If SQL references columns missing on legacy `images` (e.g. `title`, `description`, `instagram_posted`, `color_label`), add idempotent `_migrate_add_column` (or equivalent) in `init_database` before queries run, matching patterns already used for other columns.
- If `_migrate_unified_image_keys` or backup I/O raises (collision, permissions), make migration safe: log clearly, avoid aborting the whole request path when possible, or surface a deterministic error that does not mask the root cause.
- If `with_db` swallows specifics, improve logging or map known failure classes to clearer messages without breaking the API contract.

Implement the minimal code fix so the catalog list query succeeds on a normal library DB after app start. Confirm `init_database` leaves the schema compatible with `query_catalog_images`.

Verify manually or via automated request: response status **200**, JSON includes `images` (array) and `total` (non-negative integer) for a populated library.
</action>

<acceptance_criteria>
- `GET /api/images/catalog` returns HTTP **200** (not 500) against a dev environment with a configured catalog and indexed images.
- Response body includes `images` and `total` fields consistent with successful pagination.
- No uncaught exceptions in the server log for the happy path after the fix.
- If a regression test exists or is trivial to add for the catalog route or `query_catalog_images`, it passes; otherwise document the manual curl/CLI steps used in Task 3.
</acceptance_criteria>

## Task 2: Fix `CatalogTab.tsx` empty-state and error UX

<read_first>
- `apps/visualizer/frontend/src/components/images/CatalogTab.tsx` (loading branch, `noFiltersAndEmptyDb`, empty early return ~157–171, main layout with filters)
- `apps/visualizer/frontend/src/services/api.ts` (or equivalent) — how `ImagesAPI.listCatalog` surfaces HTTP errors
</read_first>

<action>
**Always render the filter bar** (and related controls that belong with it: posted/month/keyword/rating/date/color, clear, pagination when applicable) in the non-loading layout, regardless of `total`.

Replace the incorrect `noFiltersAndEmptyDb` definition (`!hasActiveFilters` alone) with logic that means: *truly empty catalog* — e.g. successful load with `total === 0` **and** no active filters **and** not in an error state. Do **not** treat API failure as “empty DB.”

Add explicit **error state**: when `ImagesAPI.listCatalog` throws or returns an error response, set state (e.g. `loadError: string | null`) and render a visible error message in the catalog area (reuse existing typography/patterns). On retry or successful reload, clear the error.

Ensure **filtered-to-zero** still shows filters and copy such as “No images match the filters” (existing string pattern), not the “empty database” message.

Refactor the early returns so `CatalogSettingsPanel` + filter row stay mounted; only the grid/summary region switches among loading, error, empty-DB, filtered-empty, and populated states.
</action>

<acceptance_criteria>
- With `total === 0` after a **failed** API call, filter controls remain **visible** and an error message is shown (not “No Catalog Images / database is empty”).
- With `total === 0` and **no** filters after a **successful** load, the dedicated empty-DB message may still appear, but filters remain visible.
- With active filters and `total === 0`, filters remain visible and mismatch copy is shown.
- `noFiltersAndEmptyDb` (or its successor) is not equivalent to `!hasActiveFilters` alone.
</acceptance_criteria>
