# Plan 04: Unify composite keys and migrate library DB foreign keys

<objective>
Make `generate_key` in `core.database` match `generate_record_key` in `lightroom.reader` (date portion `YYYY-MM-DD` only), migrate existing `images.key` and all dependent tables in the **library** SQLite so thumbnails and matches keep working, and ensure `store_image` upserts refresh the Lightroom `id` column for stable `id_local` identity.
</objective>

<tasks>

## Task 1: Unify `generate_key` with reader date truncation

<read_first>
- lightroom_tagger/core/database.py
- lightroom_tagger/lightroom/reader.py
- lightroom_tagger/core/test_database.py
</read_first>

<action>
In `lightroom_tagger/core/database.py`, change `generate_key(record: dict) -> str` so it uses the same algorithm as `lightroom_tagger.lightroom.reader.generate_record_key`:

- `date_taken = record.get('date_taken', 'unknown')`
- `date_part = date_taken[:10] if date_taken else 'unknown'`
- `filename = record.get('filename', 'unknown')`
- `return f"{date_part}_{filename}"`

Do not change `generate_record_key` in `reader.py` in this task (it should already match).
</action>

<acceptance_criteria>
- `rg -n "date_part = date_taken\\[:10\\]" lightroom_tagger/core/database.py` returns at least one match inside `generate_key`.
- `python -m pytest lightroom_tagger/core/test_database.py::TestDatabase::test_generate_key -q` exits with status code 0.
</acceptance_criteria>

## Task 4: One-time library DB migration for renamed keys

<read_first>
- lightroom_tagger/core/database.py
- lightroom_tagger/core/test_database.py
</read_first>

<action>
In `lightroom_tagger/core/database.py`, implement `_migrate_unified_image_keys(conn: sqlite3.Connection) -> None` and call it from `init_database` after existing `_migrate_add_column` calls and before `conn.commit()` of the initial setup (or immediately after schema creation in the same function before return — must run on every `init_database` open but be **idempotent**).

Idempotency: use `PRAGMA user_version` on this same library DB connection. If `user_version >= 1`, return immediately. After successful migration, execute `PRAGMA user_version = 1`.

Migration algorithm:

1. Load all rows from `images` into memory as dicts via existing row factory.

2. For each row, compute `new_key = generate_key(row)` from the **current** unified `generate_key`. If `new_key == row['key']`, skip.

3. If any two distinct existing keys would map to the same `new_key`, raise `RuntimeError` whose message contains exact substring `migrate_unified_image_keys: key collision`.

4. For each pair `(old_key, new_key)` with `old_key != new_key`, update referencing tables in this order (SQLite foreign keys may be off; still update children first):  
   `UPDATE matches SET catalog_key = ? WHERE catalog_key = ?`  
   `UPDATE rejected_matches SET catalog_key = ? WHERE catalog_key = ?`  
   `UPDATE vision_cache SET key = ? WHERE key = ?`  
   `UPDATE vision_comparisons SET catalog_key = ? WHERE catalog_key = ?`  
   `UPDATE image_descriptions SET image_key = ? WHERE image_key = ? AND image_type = 'catalog'`  
   `UPDATE instagram_dump_media SET matched_catalog_key = ? WHERE matched_catalog_key = ?`  
   Use separate executemany or loop with `commit()` every N rows — single final `commit` is acceptable.

5. Finally `UPDATE images SET key = ? WHERE key = ?` for each remap.

6. Set `PRAGMA user_version = 1`.

Add unit test in `test_database.py` named `test_migrate_unified_image_keys_rewrites_legacy_key` that: creates a temp DB with `init_database`, `PRAGMA user_version = 0`, raw `INSERT INTO images` supplying column `key` = `'2024-01-15T00:00:00_photo.jpg'`, `date_taken` = `'2024-01-15T00:00:00'`, `filename` = `'photo.jpg'`, and minimal non-null columns required by schema (`id`, `filepath`, etc. use empty string or `0` as appropriate for SQLite), then invokes the migration function directly (import `migrate_unified_image_keys` from `lightroom_tagger.core.database` — implement that as a **public** thin wrapper calling `_migrate_unified_image_keys(conn)`), then asserts `SELECT key FROM images` returns `2024-01-15_photo.jpg`.

Implement in `database.py` a public `def migrate_unified_image_keys(conn: sqlite3.Connection) -> None:` that only calls `_migrate_unified_image_keys(conn)`.
</action>

<acceptance_criteria>
- `rg -n "def _migrate_unified_image_keys" lightroom_tagger/core/database.py` returns exactly one match.
- `rg -n "def migrate_unified_image_keys" lightroom_tagger/core/database.py` returns exactly one match.
- `rg -n "PRAGMA user_version" lightroom_tagger/core/database.py` returns at least two matches (read and write).
- `rg -n "migrate_unified_image_keys: key collision" lightroom_tagger/core/database.py` returns at least one match.
- `rg -n "instagram_dump_media" lightroom_tagger/core/database.py` returns at least one match below the line defining `_migrate_unified_image_keys`.
- `rg -n "def test_migrate_unified_image_keys_rewrites_legacy_key" lightroom_tagger/core/test_database.py` returns exactly one match.
- `python -m pytest lightroom_tagger/core/test_database.py -q` exits with status code 0.
</acceptance_criteria>
