<objective>
Implement **`single_score`** and **`batch_score`** job handlers that score images per selected active perspectives using the OpenAI-compatible vision stack, validate each LLM reply with **`parse_score_response_with_retry`** and a non-`None` **`llm_fixer`** from **`make_score_json_llm_fixer`**, and persist exclusively through **`supersede_previous_current_scores`** + **`insert_image_score`**. Add **`fingerprint_batch_score`** and triplet-based checkpoint resume compatible with existing orphan recovery. Centralize domain logic in **`lightroom_tagger/core/scoring_service.py`** so handlers stay thin orchestration.
</objective>

<action>
Create **`lightroom_tagger/core/scoring_service.py`** with typed helpers (mypy-clean):

1. **`def compute_prompt_version(perspective_row: dict) -> str`** — stable string from `slug` + hash of `prompt_markdown` (UTF-8), e.g. `f"{slug}:{hashlib.sha256(...).hexdigest()[:16]}"` per `06-CONTEXT.md` D-23.

2. **`def perspective_score_already_current(conn, image_key, image_type, perspective_slug, prompt_version) -> bool`** — True when a row exists with `is_current=1` for that quadruple (SQL `SELECT 1 ... LIMIT 1`).

3. **`def delete_scores_for_version(conn, image_key, image_type, perspective_slug, prompt_version) -> None`** — used when `force=True` before re-insert (D-26).

4. **Core scoring function** (name at implementer choice, e.g. `score_image_for_perspective`) with signature along the lines of:
   - `(db, *, image_key: str, image_type: str, perspective_slug: str, force: bool, provider_id: str | None, model: str | None, log_callback: LogCallback) -> tuple[str, bool, str | None]`
   - Return format **mirrors** `_describe_single_image`: `('scored'|'skipped'|'failed', success_bool, error_message)`.

   **Inside each call:**
   - Resolve filesystem path: **catalog** → `get_image` + `resolve_filepath` + exists check (same as `describe_matched_image`); **instagram** → `get_instagram_dump_media` + `file_path` exists check (same as `describe_instagram_image`).
   - Load perspective: `get_perspective_by_slug`; fail if missing or `active != 1`.
   - Compute `pv = compute_prompt_version(row)`.
   - If not `force` and `perspective_score_already_current(...)`: return `('skipped', False, '...')` without calling the model.
   - If `force`: call `delete_scores_for_version(...)` for `(image_key, image_type, perspective_slug, pv)` then proceed (or equivalent that avoids `UNIQUE` violation on `uq_image_scores_versioned`).
   - Image prep: reuse the same pattern as `lightroom_tagger.core.analyzer._describe_image_via_provider` — `get_viewable_path`, `compress_image`, temp file cleanup `finally`.
   - **`FallbackDispatcher.call_with_fallback`**: `operation` string **`"score"`** (or reuse `"describe"` if registry lacks score — document choice in module docstring; prefer **`"score"`** if `ProviderRegistry` / tests allow).
   - `fn_factory` returns `lambda: generate_description(client, mdl, compressed_path, log_callback=log_callback, user_prompt=build_scoring_user_prompt(perspective_row))`.
   - After `raw, actual_provider, actual_model = dispatcher.call_with_fallback(...)`, re-resolve **`client = registry.get_client(actual_provider)`** and bind **`llm_fixer = make_score_json_llm_fixer(functools.partial(complete_chat_text, client, actual_model), ...)`** so JSON repair uses the same provider that produced `raw`.
   - Call **`parse_score_response_with_retry(raw, llm_fixer=llm_fixer)`** — must **not** pass `llm_fixer=None` in production path.
   - Validate `parsed.perspective_slug == perspective_slug` (after strip); on mismatch treat as failure with clear message (prevents wrong-key inserts).
   - Transaction: `supersede_previous_current_scores(db, image_key, image_type, perspective_slug, pv)` then `insert_image_score` with dict keys: `image_key`, `image_type`, `perspective_slug`, `score`, `rationale`, `model_used` (`f"{provider}:{model}"` or model only), `prompt_version`=`pv`, `scored_at` ISO-8601 UTC, `is_current=1`, `repaired_from_malformed` from repaired flag.
   - On `StructuredOutputError`: return `('failed', False, exc message suitable for job log)` without persisting.

5. **Exports:** `__all__` listing public functions used by handlers.
</action>

<action>
1. Import **`fingerprint_batch_score`** from `.checkpoint`.

2. **`def handle_single_score(runner, job_id: str, metadata: dict)`**
   - Validate `image_key`, `image_type` in `('catalog','instagram')`, non-empty `perspective_slugs` list (or default to all active perspectives if metadata omits — **prefer requiring explicit list** to match describe jobs).
   - Open library DB via `init_database` + `LIBRARY_DB` / config pattern (copy `handle_single_describe`).
   - Loop slugs sequentially; call scoring_service function; `complete_job` with summary counts on full success; `fail_job` if any requested slug hard-fails (or partial success policy: document chosen policy in handler docstring — **default: fail job if any slug fails** for clarity).

3. **`def handle_batch_score(runner, job_id: str, metadata: dict)`**
   - Build **work list**: Cartesian product of target images (reuse **same SQL selection** as `handle_batch_describe` for catalog + instagram keys) × **each selected `perspective_slug`**.
   - Compute `fp = fingerprint_batch_score(metadata, sorted_triples)`.
   - Resume: read `metadata.checkpoint` when `job_type == 'batch_score'` and fingerprint matches; subtract `processed_triplets` set from work list (same mismatch log style as batch_describe).
   - **Parallelism:** if `max_workers > 1` and enough units, use `ThreadPoolExecutor` like `handle_batch_describe` but worker calls scoring_service with **worker-owned** `sqlite3` connection.
   - **Checkpoint:** after each completed triplet, append `f"{key}|{itype}|{slug}"` to sorted set, call `runner.persist_checkpoint` with `checkpoint_version` 1 envelope via runner helper (same as other handlers).
   - Respect `runner.is_cancelled`, `finalize_cancelled`, consecutive failure cap **consistent with batch_describe** (reuse constants / pattern).
   - `clear_checkpoint` on successful completion.

4. Register in **`JOB_HANDLERS`**: `'single_score': handle_single_score`, `'batch_score': handle_batch_score`.
</action>

<threat_model>
**ASVS L1 — scoring execution**

- **Threat:** SQL injection via slugs — mitigate with parameterized queries only in `scoring_service` / DB helpers (already the pattern).
- **Threat:** Unbounded LLM output — rely on `STRUCTURED_OUTPUT_MAX_CHARS` in structured_output (Phase 5); scoring prompts should stay concise.
- **Threat:** Checkpoint metadata growth — enforce max processed entries like `_CHECKPOINT_MAX_ENTRIES` in `handle_batch_score` (reuse constant from handlers.py).
- **Privacy:** Score rationale may describe image content; treat like descriptions (local deployment assumption unchanged).
</threat_model>
