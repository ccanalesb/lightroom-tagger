<objective>
Fix three compounding failures that cause batch vision/description jobs to waste thousands of API calls on oversized images: add `.sr2` (Sony Raw 2) to RAW format support, validate cached image size before storing, and pre-filter candidates by file size before entering the batch API pipeline. Must run before all other Phase 4 plans — batch describe (04-04) and provider health (04-05) depend on a reliable pipeline.
</objective>

<context>
Discovered during job `17975aa0` (vision_match, year=2025, gpt-5-mini via github_copilot):
- 8,346 warnings total: 4,110 "single-item chunk still too large" + 4,236 "413 payload too large, splitting"
- Root cause: ~70 `.sr2` files (Sony Raw 2, ~20MB each) bypass RAW conversion because `.sr2` is not in `RAW_EXTENSIONS`
- `get_or_create_cached_image` stores the original 20MB path when compression fails (line 93-96)
- Each oversized candidate triggers ~4 wasted API calls (10→5→2→1→fail) before being skipped
- Working candidates have cached sizes of 50-135KB; failing ones are 20MB (cached == original)

Related prior work:
- `docs/plans/2026-04-10-001-feat-reactive-vision-error-handling-plan.md` — reactive error handling (PayloadTooLargeError, chunk halving, adaptive max_tokens) — all implemented
- `docs/plans/2026-03-26-cache-compressed-images.md` — vision cache implementation
</context>

### Task 2: Add max size validation to cached image storage
**Type:** modify
**Files:** `lightroom_tagger/core/vision_cache.py`

<read_first>
- lightroom_tagger/core/vision_cache.py (`get_or_create_cached_image`, lines 48-112)
- lightroom_tagger/core/config.py (for adding a config key if needed)
</read_first>

<action>
1. Define `MAX_CACHED_IMAGE_KB = 512` at module level (512KB — working cached images are 50-135KB; anything above 500KB indicates failed compression).
2. At line 93, where `temp_path == viewable_path == original_path` (neither conversion nor compression worked), check `os.path.getsize(original_path) / 1024 > MAX_CACHED_IMAGE_KB`. If oversized, return `None` instead of storing and returning the original path. This signals to callers that the image is unusable for vision APIs.
3. Also add a size check after successful compression (line 98-103): if the compressed file still exceeds `MAX_CACHED_IMAGE_KB`, remove it and return `None`.
4. In both cases, store a sentinel in the DB cache so future calls don't re-attempt compression. Use the existing `store_vision_cached_image` with `compressed_path=None` or a special marker path like `__oversized__`.
</action>

<acceptance_criteria>
- `rg "MAX_CACHED_IMAGE_KB" lightroom_tagger/core/vision_cache.py` returns a match.
- `rg "oversized" lightroom_tagger/core/vision_cache.py` returns at least one match.
- When `get_or_create_cached_image` is called with a 20MB file that can't be compressed, it returns `None`.
</acceptance_criteria>
