<objective>
Add durable SQLite storage for exact evaluated comparison pools at match time (`.planning/phases/19-generate-read-only-html-comparison-pool-report/19-CONTEXT.md` **D-01**, **D-03**). Deliver idempotent migration, bootstrap DDL for greenfield DBs, and a typed insert/fetch API in `match_pool_snapshots.py` without touching product UI or match write semantics beyond new INSERTs.
</objective>

<decision_coverage>
- **D-01:** Exact pool persistence via snapshot rows tied to each capture run (append-only `snapshot_id`; report consumers take latest by `captured_at` unless filtered by `source_job_id` — see 19-03).
- **D-03:** New normalized tables are the durable pool-capture contract planned in CONTEXT.
- **D-02, D-04–D-14:** Deferred to later plans; no UX or CLI in this plan.
</decision_coverage>

<threat_model>
**ASVS level:** 1  
**Block on severity:** high  

**Assets:** Library DB contents; future HTML will embed captions and reasoning (handled in 19-03); optional `debug_resolved_path` stores absolute paths for debug-only display.

**Threats and mitigations (this plan):**
- **T-19-schema Injection / fragile DDL:** Use parameterized Python APIs only for row inserts in `match_pool_snapshots.py`; static DDL lives in migration + bootstrap strings executed via `executescript` (no string-built DDL from user input).
- **T-19-data integrity / mistaken writeback:** `insert_comparison_pool_snapshot` performs **INSERT** only into new tables — must not call `store_match`, `mark_dump_media_processed`, or Lightroom writers.
- **T-19-path retention:** Column `debug_resolved_path` exists **only** to support D-08/D-13 in the report’s hidden debug layer; primary HTML is out of scope here but schema name documents intent.

**Residual risks (not blocking L1):** Large snapshot volume per run — operators mitigate via match batching; no hard DB cap in this plan.
</threat_model>
