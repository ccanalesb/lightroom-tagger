# Plan 10-02: Suggestions `offset` + catalog-only scores for identity aggregation

## Intent

1. **`/api/identity/suggestions`:** `identity.py` parses `offset` via `_clamp_pagination` but assigns it to `_offset` and never passes it to `suggest_what_to_post_next` (D-03). The service slices with `candidates_full[:limit]` only (D-03).
2. **Response shape:** Return `total` = number of suggestion candidates **before** applying `offset`/`limit`, matching `best-photos` (`items` + `total` + `meta`) so clients can paginate (D-04).
3. **Identity aggregation:** `_SCORES_BASE_SQL` currently pulls all `image_type` values; `compute_image_aggregate_scores` buckets by `image_key` only, so a catalog key and an Instagram `media_key` string collision would merge rows (D-05). Per Phase 8 D-40 / 10-CONTEXT, restrict base SQL to **`s.image_type = 'catalog'`** so all identity features (best photos, fingerprint, suggestions) use catalog scores only.

## must_haves (goal-backward)

1. `suggest_what_to_post_next(..., limit=L, offset=O)` returns candidates corresponding to `candidates_full[O:O+L]` after the same sort order currently implemented.
2. JSON from `GET /api/identity/suggestions` includes top-level **`total`** (integer, full candidate count before pagination).
3. `_SCORES_BASE_SQL` contains **`AND s.image_type = 'catalog'`** (or equivalent single predicate) so Instagram score rows are excluded from identity aggregation SQL.
4. `uv run pytest lightroom_tagger/core/test_identity_service.py apps/visualizer/backend/tests/test_identity_api.py -q` exits 0.

## Verification criteria

- `rg "s.image_type = 'catalog'" lightroom_tagger/core/identity_service.py` matches within `_SCORES_BASE_SQL`.
- `rg "offset" apps/visualizer/backend/api/identity.py` shows `suggest_what_to_post_next` call passes `offset=offset` (variable name may be `offset` after unpack).
- `rg '"total"' apps/visualizer/backend/api/identity.py` or response assembly includes `total` in `jsonify` for suggestions — verify by reading `suggestions()` after edit.
- `pytest` commands above exit 0.

<threat_model>

**ASVS L1:** Changes are read-only API behavior over the library DB; `limit` and `offset` stay clamped via `_clamp_pagination`. Catalog-only SQL reduces cross-type data mixing, not expands it. **Risks:** negligible. **Mitigations:** same pagination helper as `best-photos`; no new raw SQL from request strings.

</threat_model>
