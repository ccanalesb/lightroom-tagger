# PLAN: 260427-d5m Embed job log observability quick fix

## Overview

Runtime inspection showed three practical observability gaps:
1) quick inspector reports zero logs because it still reads legacy `jobs.logs` instead of `job_logs`,
2) jobs list/active payloads hide useful log signal (`logs_total`, warning/error counts),
3) large checkpoint arrays in API responses drown out useful log context and inflate payload size.

This quick task improves observability without changing core job execution behavior.

## Tasks

### Task 1: Align inspect tool with current log storage
- files:
  - `.cursor/skills/investigate-job/scripts/inspect_job.py`
- action:
  - Read logs from `job_logs` table instead of legacy `jobs.logs`.
  - Report checkpoint processed key counts using modern embed checkpoint keys (`processed_pairs`) in addition to legacy variants.
  - Keep output shape backward compatible (job fields + metadata + last log entries).
- verify:
  - Run inspector for a known running job and confirm non-zero log entries appear.
- done:
  - Inspector reflects real-time persisted logs and checkpoint progress.

### Task 2: Expose lightweight log summaries on list/active surfaces
- files:
  - `apps/visualizer/backend/database.py`
  - `apps/visualizer/backend/api/jobs.py`
  - `apps/visualizer/backend/tests/test_database.py`
  - `apps/visualizer/backend/tests/test_jobs_api.py`
- action:
  - Add DB helper(s) to compute per-job log summary (`logs_total`, `warning_count`, `error_count`, `last_log_at`) without loading full logs.
  - Attach these summary fields to rows returned by `list_jobs` / `get_active_jobs`.
  - Ensure jobs API list/active routes return those summary fields.
- verify:
  - Run backend tests for database/jobs API.
  - Confirm `/api/jobs/` and `/api/jobs/active` include summary counters for the running embed job.
- done:
  - Queue surfaces can show whether logs exist and whether warnings/errors are growing.

### Task 3: Compact checkpoint payloads in API responses
- files:
  - `apps/visualizer/backend/api/jobs.py`
  - `apps/visualizer/backend/tests/test_jobs_api.py`
- action:
  - Sanitize API job payloads by replacing large checkpoint key lists (`processed_pairs`, `processed_media_keys`, etc.) with count fields (e.g., `processed_pairs_count`).
  - Apply compaction to list, active, and detail responses while leaving DB-stored metadata untouched.
- verify:
  - Confirm job detail still includes checkpoint metadata but omits huge key arrays and includes counts.
  - Run jobs API tests including new coverage for compaction.
- done:
  - Job payload size drops significantly and log sections remain easy to inspect.
