# Quick Task 260420-840: Consolidate image metadata display across frontend views - Context

**Gathered:** 2026-04-20
**Status:** Ready for research

<domain>
## Task Boundary

Consolidate how image metadata (scores, identity perspectives, rating, pick,
posted flags, etc.) is displayed across the visualizer frontend. Today, the
same image renders with different metadata in each of these surfaces:

- **Insight → Top Photos** (`components/insights/TopPhotosStrip.tsx`, opens
  `CatalogImageModal`)
- **`/images` → Instagram tab** (`components/images/InstagramTab.tsx` +
  `components/instagram/*`)
- **`/images` → Catalog tab** (`components/images/CatalogTab.tsx` +
  `components/catalog/CatalogImageCard.tsx` + `CatalogImageModal.tsx`)
- **`/catalog`** (currently folded into `/images?tab=catalog`; same
  CatalogImageCard)
- **`/analytics` → Not posted** (`components/analytics/UnpostedCatalogPanel.tsx`,
  reuses `CatalogImageCard` + `CatalogImageModal` but with a zeroed stub)
- **`/identity` → Best photos / Post-next** (`components/identity/BestPhotosGrid.tsx`
  + `PostNextSuggestionsPanel.tsx`)

Mobile view is currently not accounted for — modals and tile density assume
desktop widths.

</domain>

<decisions>
## Implementation Decisions

### Scope — "Consolidate" means (Q0)
**Full data + UI consolidation (Q0: option 3).**

Normalize the underlying types (`CatalogImage`, `UnpostedCatalogItem`,
`IdentityBestPhotoItem`, Instagram image shapes) to a single canonical
frontend type, AND share the UI components across every site. This fixes
the root cause of today's drift: the `bestPhotoToCatalogStub` /
`unpostedRowToCatalogStub` adapters that zero-out score fields before
opening the modal.

### Canonical type shape (Q1)
**`ImageView` = superset type (Q1: option 1).**

One type holds every possible metadata field:
- Identity: `key`, `image_type` (`'catalog' | 'instagram'`), `filename`,
  `date_taken`, `filepath`
- Lightroom: `rating`, `pick`, `color_label`, `keywords`, `title`, `caption`
- Posted state: `instagram_posted`
- AI / analysis: `ai_analyzed`, `description_best_perspective`,
  `description_perspectives`
- Catalog scoring: `catalog_perspective_score`, `catalog_score_perspective`
- Identity scoring: `aggregate_score`, `perspectives_covered`,
  `per_perspective: IdentityPerPerspectiveScore[]`

Fields are optional/nullable when not applicable. Adapter functions at the
API-response boundary (`fromCatalogListRow`, `fromUnpostedRow`,
`fromBestPhotoRow`, `fromInstagramRow`, `fromImageDetail`) normalize
server responses into `ImageView`.

### Modal data strategy (Q2)
**Always re-fetch full detail on modal open (Q2: option 1).**

List endpoints stay lean (tile-relevant fields only). When a tile is
clicked, the shared modal calls a new detail endpoint that returns the
full `ImageView` including identity `per_perspective` breakdown. This
guarantees the modal is identical regardless of entry point and fixes
the "clicked from Unposted and scores are zero" bug at the root.

**Implication:** a new backend endpoint is required. Today the backend
exposes `/api/images/<type>/<key>/thumbnail` but no detail endpoint.
The research phase should recommend the endpoint shape
(e.g. `GET /api/images/<type>/<key>` returning the full `ImageView`
superset, including identity scores joined in).

### Badge taxonomy (Q3)
**Core badges (always shown when the field is present):**
- Rating stars (★)
- Pick flag
- Posted to Instagram
- AI-analyzed
- **Primary score pill** — single pill, context-driven source:
  - Identity surfaces → `aggregate_score` with "Aggregate" label
  - Catalog surfaces / Unposted → `catalog_perspective_score` with the
    perspective slug (fall back to `description_best_perspective` score
    if no persisted catalog score)
  - Instagram tiles → no score pill (Instagram images don't have these)

**Modal-only (always shown in every modal, regardless of entry point):**
- **Identity per-perspective breakdown** (`per_perspective[]`) —
  compact pill row or mini-table showing each perspective's score
- `aggregate_score` number (alongside the breakdown, with
  `perspectives_covered` context)
- Description best-perspective score detail
- Full keyword list, title, caption, copyright

**Context-only (only on the surface that needs it):**
- None — "Identity perspectives_covered" was moved out of
  context-only; it's now part of the modal-core alongside
  the per-perspective breakdown.

### Mobile strategy (Q4)
**Responsive single-component approach (Q4: option a).**

- Modal becomes a **full-screen sheet below `md` (768px)** and a
  standard centered modal at `md+`. One component, two layouts via
  Tailwind responsive classes.
- Tile grids keep the responsive `grid-cols-{n}` pattern but all sites
  adopt the same breakpoints so density is consistent.
- Metadata badges wrap cleanly on narrow widths — no horizontal
  scroll; any overflow goes to a "+N more" affordance or stacked rows.

### Claude's Discretion
- Exact file paths and module boundaries for the new shared components
  (`<ImageView>`, `<ImageMetadataBadges>`, `<ImageTile>`,
  `<ImageDetailModal>`) — planner/researcher will propose.
- Whether Instagram images get their own lightweight `ImageView` variant
  or share the full superset with most fields empty — research will
  decide based on how divergent the Instagram modal content actually is.
- Whether adapters live in `services/api.ts` alongside existing types,
  or a new `utils/imageView.ts` module.
- Whether to fold `TopPhotosStrip`'s custom pill styling into the shared
  badge component or keep strip-specific tile styling (only the modal
  contents must consolidate — strip-vs-grid-vs-card tile shapes can
  still differ, as long as the badge components inside them are
  identical).

</decisions>

<specifics>
## Specific Ideas

**Known current drift points** (to verify and fix):

1. `TopPhotosStrip.tsx` uses inline `<span>` pills with `bg-accent-light`
   for score and `rounded-full border` for perspective count.
2. `CatalogImageCard.tsx` mixes the `Badge` component with inline
   `<span>` pills (three distinct pill styles in one card).
3. `bestPhotoToCatalogStub` (TopPhotosStrip) and `unpostedRowToCatalogStub`
   (UnpostedCatalogPanel) zero out score/perspective fields before
   handing to the modal — modal then shows "no score" even though the
   image has one.
4. `CatalogImageModal` (408 lines) and `ImageDetailsModal` (Instagram,
   159 lines) are two separate modals with overlapping concerns.
5. Mobile: `CatalogImageModal` uses desktop-assumed sizing. Not verified
   on narrow viewports.

**Known data locations** (from quick grep):
- `IdentityAPI.getBestPhotos` already returns `per_perspective[]` +
  `aggregate_score` per image.
- `ImagesAPI.listCatalog` returns `catalog_perspective_score`,
  `description_best_perspective`, `description_perspectives`.
- `AnalyticsAPI.getUnpostedCatalog` returns minimal fields — the stub
  zeroes the rest. This endpoint likely needs widening OR the new
  detail endpoint becomes the source of truth for modal data.

</specifics>
